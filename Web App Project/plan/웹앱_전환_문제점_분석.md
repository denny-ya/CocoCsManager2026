## 🔴 치명적 문제 (구현 자체가 막힐 수 있음)

### 1. GAS에는 폴더 구조가 없음

계획서의 폴더 구조(`css/`, `js/pages/` 등)는 GAS에서 **불가능**합니다.
GAS 스크립트 에디터는 파일을 모두 **루트 단일 레벨**로만 관리하며, `css/styles.css` 같은 경로로 파일을 참조할 수 없습니다.

**대응 방안**
- 모든 `.html` 파일은 같은 레벨에 위치해야 함
- CSS/JS는 `<?!= HtmlService.createHtmlOutputFromFile('styles')?>` 방식으로 인라인 포함

---

### 2. 배차·영업점·업무 가이드 API 누락

계획서 1-4절 API 목록에는 5개 API만 나와 있지만, 화면 목록에는 다음 3개 화면이 추가로 존재합니다:

| 화면 | API 언급 여부 |
|------|-------------|
| 배차 목록 | ❌ 없음 |
| 영업점 주소 검색 | ❌ 없음 |
| 업무 가이드 | ❌ 없음 |

**대응 방안**
- 각 화면의 데이터 소스 명확화 필요 (GAS 함수? Sheets 직접 조회? 정적 데이터?)

## 🟠 구조적 문제 (나중에 크게 고생할 수 있음)

### 3. `google.script.run` 콜백 지옥

Phase 5에서 `fetch` → `google.script.run`으로 전환 예정이지만, `google.script.run`은 **Promise를 지원하지 않고 콜백 기반**입니다.
여러 API를 순차/병렬 호출할 때 코드가 급격히 복잡해집니다.

```js
// 콜백 중첩 예시 — 유지보수 어려움
google.script.run
  .withSuccessHandler(function(result) {
    google.script.run
      .withSuccessHandler(function(result2) {
        // ...
      })
      .anotherFunc();
  })
  .searchBS(keyword);
```

**대응 방안**
- `google.script.run`을 Promise로 감싸는 래퍼를 Phase 3에서 초기 설계

```js
function runScript(fnName, ...args) {
  return new Promise((resolve, reject) => {
    google.script.run
      .withSuccessHandler(resolve)
      .withFailureHandler(reject)
      [fnName](...args);
  });
}
```

### 4. 상태 관리 전략이 너무 모호함

계획서에서 "전역 변수 또는 간단한 상태 객체"로만 언급되어 있습니다.
SPA에서 전역 변수 남용은 화면 전환 시 **상태 오염**, **데이터 꼬임**으로 이어집니다.

**대응 방안**
- 최소한 단방향 상태 패턴 설계 필요 (앱 전체 state 객체 + 렌더 함수)
- Phase 3 공통 컴포넌트 작업 시 함께 설계

---

### 5. 보안 모델 누락

`google.script.run`은 웹앱 **배포자의 권한**으로 실행됩니다.
접근 권한을 "전체 공개"로 설정하면 URL을 아는 누구나 데이터 변경 함수를 호출할 수 있습니다.

| 위험 함수 | 영향 |
|----------|------|
| `saveMemo()` | 임의 메모 저장 |
| `markAsComplete()` | 임의 점검 완료 처리 |

**대응 방안**
- 접근 권한을 "조직 내 사용자"로 제한하거나
- 서버 함수 내에서 `Session.getActiveUser()`로 호출자 검증 로직 추가

## 🟡 성능·UX 문제

### 6. GAS 콜드 스타트 시간 미반영

| 지연 유형 | 예상 시간 |
|----------|---------|
| GAS 웹앱 최초 요청 콜드 스타트 | 2~5초 |
| `google.script.run` 개별 호출 | 1~3초 |

계획서 위험 요소에 "초기 로딩 느림"만 언급되어 있고, **매 API 호출마다 발생하는 지연**은 다루지 않습니다.
검색처럼 빈번한 호출이 있는 화면에서 UX가 상당히 나빠질 수 있습니다.

**대응 방안**
- 호출 결과 클라이언트 캐싱 (sessionStorage 활용)
- 스켈레톤 로딩 UI를 모든 API 호출 화면에 적용

---

### 7. GAS 일일 할당량 구체적 수치 없음

계획서에 "할당량 초과" 위험은 언급했지만 실제 한도가 명시되지 않았습니다.

| 항목 | 한도 |
|------|------|
| 스크립트 일일 실행 시간 | 6시간/일 (개인·Workspace 공통) |
| 단일 실행 최대 시간 | **6분** (초과 시 강제 종료) |
| URL Fetch 호출 | 20,000회/일 (개인 계정) |

동시 사용자 수를 고려한 할당량 여유 분석이 필요합니다.

## 🔵 개발 운영 문제

### 8. 버전 관리(Git) 전략 없음

`clasp`가 "선택" 사항으로 분류되어 있지만, GAS 웹 에디터만으로는 다음이 어렵습니다:

- 코드 히스토리 관리
- 버그 발생 시 이전 버전 롤백
- 협업 및 코드 리뷰

**대응 방안**
- `clasp` + Git을 필수 개발 환경으로 격상
- GitHub 또는 사내 저장소와 연동

---

### 9. 배포 URL 공유 방안 없음

GAS 웹앱 URL 형태: `https://script.google.com/macros/s/[긴-ID]/exec`

사용자가 기억하거나 입력하기 어려운 URL입니다. 배포 후 사용자에게 어떻게 전달할지 계획이 없습니다.

**대응 방안 (택1)**
- URL 단축 서비스 활용
- QR 코드 생성 후 사내 공유
- 사내 포털/인트라넷에 링크 등록

---

### 10. PWA 설정이 "선택"으로 처리됨

모바일 앱을 웹으로 전환하는 만큼, 홈 화면 추가(Add to Home Screen)는 사용성 측면에서 중요합니다.
선택 사항으로 두면 기존 앱 대비 접근성이 낮아질 수 있습니다.

**대응 방안**
- Web App Manifest 최소 설정은 Phase 6에 필수 항목으로 포함
