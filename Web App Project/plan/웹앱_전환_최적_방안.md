# 웹앱 전환 최적 방안 및 설계 방향

> **작성일**: 2026-03-12

## 🏆 최적의 구현 방안: "React (Vite) 기반 Single-File GAS 웹앱 아키텍처"

프론트엔드를 최신 **React (Vite) 환경에서 개발**하고, 배포 직전에 빌드 플러그인을 사용하여 모든 컴포넌트, JS 로직, CSS를 **단일 `index.html` 파일로 압축**하여 GAS 서버에 배포(`clasp` 활용)하는 전략입니다.

이 구조는 구글 앱스 스크립트(GAS) 환경에서 복잡한 웹앱을 구축할 때 가장 권장되는 마스터 패턴입니다.

### 💡 방안이 훌륭한 이유 (문제점 완벽 해결)

1. **GAS의 폴더 구조 불가 제약 우회 (🔴 치명적 문제 해결)**
   - 로컬에서는 컴포넌트, 페이지, 모델 등을 자유롭게 폴더로 나누어 쾌적하게 개발합니다.
   - 빌드 시 `vite-plugin-singlefile` 도구가 외부 CSS/JS 파일 분리 문제를 없애고, 구글이 요구하는 **루트 레벨의 단일 `index.html`로 자동 병합**해 줍니다.
2. **React Native(기존 앱) 개발 경험 및 자산 재활용**
   - 기존 RN의 컴포넌트 분리 방식, 상태 관리(State/Hook), 비즈니스 로직 등을 거의 동일하게 웹 프론트엔드 환경(React)으로 가져올 수 있습니다. Vanilla JS로 회귀할 필요가 없습니다.
3. **콜백 지옥의 원천 차단 (🟠 구조적 문제 해결)**
   - 최신 JS/React 환경이므로, 내부적으로 `google.script.run`을 `async/await` 기반의 Promise 패턴으로 감싸는 단일 유틸리티 모듈을 만들 수 있습니다. 이를 통해 비동기 호출을 깔끔하게 관리합니다.
4. **견고한 상태 관리와 SPA 라우팅 지원 (🟠 구조적 문제 해결)**
   - 전역 변수 오염 문제 없이 React Router (`HashRouter`)와 Context API(또는 Zustand/Redux)를 통해 데이터 흐름을 안전하게 통제할 수 있습니다.
5. **형상 관리 및 배포 자동화 (🔵 시스템/운영 문제 해결)**
   - Node.js 기반의 로컬 프로젝트이므로 Git 버전 관리가 자연스럽습니다.
   - CLI 도구인 `clasp`를 통해 터미널 명령어(`clasp push`) 한 번으로 1초 만에 최신 코드를 GAS에 자동 배포하는 파이프라인을 완성할 수 있습니다.

> **※ PWA에 대한 보완 (문제점 분석 10번 관련)**
> GAS 웹앱은 구글의 보안 Iframe(`*.googleusercontent.com`) 내부에서 실행되므로, 완벽한 오프라인 PWA 구동과 '홈 화면 추가 팝업' 구현이 매우 까다롭습니다. 홈 화면 추가가 강력한 요구사항이라면 프론트엔드 호스팅(Vercel/Netlify)과 GAS 백엔드(doGet API 전용)를 완전 분리하는 구조가 이상적입니다. 하지만 현재의 빠른 전환 목표에는 React-in-GAS가 일정과 효율 면에서 가장 적합합니다.

## 🗺 설계 순서 및 진행 방향 (Roadmap)

### Step 1: 개발 환경 통일 및 빌드 파이프라인 구축
- **로컬 프로젝트 세팅**: `npm create vite@latest`로 React(TypeScript/JavaScript) 프로젝트 생성.
- **Single-File 플러그인 연동**: `vite-plugin-singlefile`을 Vite 설정에 추가하여 빌드 결과물이 오직 `index.html` 하나로 출력되도록 구성.
- **Git & Clasp 초기화**: 로컬 폴더에 Git 레포지토리를 만들고, `.clasp.json` (기존 GAS 스크립트 아이디)을 연결. 터미널 명령어 한 번으로 GAS 배포가 가능한 파이프라인 완성.

### Step 2: 프론트-백엔드 코어 브릿지 모듈 설계
- **API Wrapper 작성**: `google.script.run`을 Promise로 캡슐화한 API 서비스 레이어(`api.js` 또는 `api.ts`) 설계. 모든 컴포넌트에서 비동기(`await`)로 서버 함수를 호출할 수 있게 설정.
- **데이터 소스 확정 및 Mocking**: 누락되었던 배차, 가이드 등의 데이터 소스를 확정. GAS 연동 전 UI를 테스트할 수 있는 가짜 데이터(Mock) 환경 마련.

### Step 3: 보안 통제 및 UX 안정성 확보
- **권한 확인 로직 추가**: GAS의 `Code.gs`에서 노출될 주요 함수(`saveMemo`, `markAsComplete` 등) 최상단에 `Session.getActiveUser().getEmail()` 코드를 추가하여 인가된 사용자만 접근하도록 보안 강화.
- **전역 로딩/스켈레톤 UI 구축**: GAS 함수 초기 호출 시 발생하는 콜드 스타트 지연(2~5초)을 사용자에게 부드럽게 감출 수 있는 스켈레톤 UI 및 전역 로딩 인디케이터 기획/구현.

### Step 4: 컴포넌트 마이그레이션 (React Native → React Web)
- **로컬 라우팅 시스템**: React Router (단방향 해시 라우팅: `HashRouter`) 적용.
- **디자인 시스템**: `Colors.ts`에 정의된 브랜드 컬러를 전역 CSS 변수(또는 Tailwind CSS)로 이식.
- **UI 마이그레이션**: Header, MenuCard 등 RN에서 사용했던 공통 컴포넌트의 논리적 구조를 유지한 채 순수 웹 컴포넌트로 전환 (홈 → 검색 → 실적 통계 → 가이드 순서로 마이그레이션).

### Step 5: 통합 연동 및 모바일 사용성 최적화
- **실제 API 연동**: GAS 실제 함수와 React Web App을 연결 및 테스트.
- **모바일 퍼스트 레이아웃**: 디바이스의 제한된 화면에 맞도록 레이아웃 검증 (CSS Safe Area `env(safe-area-inset-*)` 설정).
- **성능 최적화**: 검색 필터링 최적화, 빈번한 데이터의 브라우저 로컬 Cache (`sessionStorage`) 활용을 통해 API 반복 호출 방어 및 GAS 일일 할당량(Quota) 초과 방지.

### Step 6: 배포 및 운영 가이드 확립
- **접근성 개선**: URL 단축 서비스(`bit.ly` 등)를 적용하여 사용자들이 복잡한 GAS URL 대신 쉬운 주소로 접속하도록 개선.
- **운영 프로세스 문서화**: Git Commit 규칙, Clasp 배포 명령어 등 유지보수를 위한 운영 매뉴얼 정립.
